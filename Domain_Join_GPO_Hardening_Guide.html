Domain Join & Incremental Cryptography Policy Guide

Step-by-step procedure for joining a standalone workstation to the domain, isolating it in its own OU, disabling the FIPS flag at the OU level, and incrementally applying compensating cryptography policies with software testing checkpoints after each group.

1
Join the Workstation to the Domain
Performed on the workstation itself — requires local admin and domain credentials
⚠️
Before you begin Confirm DNS on the workstation is pointing to the Domain Controller. If the workstation cannot resolve the domain name, the join will fail regardless of credentials. Run nslookup corp.example.com to verify.
1
Open System Properties
Press Win + R → type sysdm.cpl → Enter. Or: right-click This PC → Properties → Change settings.
2
Click Change
On the Computer Name tab click Change…
3
Select Domain and enter the domain name
Under Member of, select Domain and type your domain FQDN (e.g. corp.example.com). Click OK.
4
Enter domain credentials
When prompted, enter a domain account that has permission to join computers (e.g. corp\administrator). Click OK.
5
Restart when prompted
Click OK through the welcome dialog, then Close, then Restart Now. The workstation will reboot and appear in the Computers container in ADUC.
ℹ️
Run from an elevated PowerShell session on the workstation. Adjust the variables at the top before running.
PowerShell — Run on Workstation
# ── Variables ────────────────────────────────────────────────────────────────
$domain     = "corp.example.com"
$ouPath     = "OU=Computers,DC=corp,DC=example,DC=com"  # initial landing OU (optional)
$credential = Get-Credential "corp\administrator"         # prompts for password

# ── Verify DNS resolves the domain before attempting join ─────────────────────
try {
    Resolve-DnsName $domain -ErrorAction Stop | Out-Null
    Write-Host "✔ DNS resolution OK for $domain" -ForegroundColor Green
} catch {
    Write-Error "DNS cannot resolve $domain. Fix DNS before joining."; exit 1
}

# ── Join the domain ───────────────────────────────────────────────────────────
Add-Computer `
    -DomainName   $domain `
    -Credential   $credential `
    -OUPath       $ouPath `     # omit this line to land in default Computers container
    -Restart:$false `
    -Force

Write-Host "✔ Domain join complete. Rebooting in 10 seconds..." -ForegroundColor Green
Start-Sleep 10
Restart-Computer -Force
⚠️
The -OUPath parameter places the computer directly into a target OU on join. If the FIPS-Exempt OU doesn't exist yet (it won't until Phase 2), omit that line — the computer lands in the default Computers container and you move it in Phase 2.
2
Create the OU, Copy Domain GPO, and Link to OU
Performed on the Domain Controller — requires Domain Admin or delegated GPO rights

Part A — Create the OU and move the workstation (ADUC)

1
Open Active Directory Users and Computers
Win + R → dsa.msc
2
Create the new OU
Right-click your domain root (or an existing parent OU) → New → Organizational Unit. Name it FIPS-Exempt-Workstations. Check Protect container from accidental deletion. Click OK.
3
Move the workstation into the new OU
In the Computers container (or wherever the workstation landed), right-click the computer object → Move… → select FIPS-Exempt-Workstations → OK.

Part B — Copy GPO and link to OU (GPMC)

4
Open Group Policy Management Console
Win + R → gpmc.msc
5
Copy the source domain GPO
Navigate to Group Policy Objects. Right-click the GPO you want to use as your baseline (e.g. Default Domain Security Policy) → Copy. Right-click Group Policy Objects → Paste. Name the new copy FIPS-Exempt-Workstation-Policy. Select Preserve existing permissions.
6
Link the new GPO to the workstation OU
Right-click FIPS-Exempt-Workstations in the tree → Link an Existing GPO… → select FIPS-Exempt-Workstation-Policy → OK.
7
Check inheritance and enforce if needed
Click the OU → Group Policy Inheritance tab. If a parent GPO is enforcing FIPS and will override your new OU policy, right-click the link for your new GPO → Enforced to ensure the OU-level setting wins.
PowerShell — Run on Domain Controller
Import-Module ActiveDirectory
Import-Module GroupPolicy

# ── Variables — adjust to your environment ───────────────────────────────────
$ouName        = "FIPS-Exempt-Workstations"
$parentOU      = "OU=Workstations,DC=corp,DC=example,DC=com"
$workstation   = "DESKTOP-EXEMPT01"        # computer name (no $)
$sourceGPO     = "Default Domain Security Policy"
$newGPOName    = "FIPS-Exempt-Workstation-Policy"

# ── Step 1: Create the OU ─────────────────────────────────────────────────────
New-ADOrganizationalUnit `
    -Name        $ouName `
    -Path        $parentOU `
    -Description "Workstations exempt from FIPS — compensating controls applied" `
    -ProtectedFromAccidentalDeletion $true

$ouDN = "OU=$ouName,$parentOU"
Write-Host "✔ OU created: $ouDN" -ForegroundColor Green

# ── Step 2: Move the workstation computer object into the new OU ──────────────
$computer = Get-ADComputer -Identity $workstation
Move-ADObject -Identity $computer.DistinguishedName -TargetPath $ouDN
Write-Host "✔ $workstation moved to $ouDN" -ForegroundColor Green

# ── Step 3: Copy the domain GPO ───────────────────────────────────────────────
$srcGPO = Get-GPO -Name $sourceGPO
$newGPO = Copy-GPO -SourceGuid $srcGPO.Id -TargetName $newGPOName
Write-Host "✔ GPO copied. New GUID: $($newGPO.Id)" -ForegroundColor Green

# ── Step 4: Link the GPO to the new OU ───────────────────────────────────────
New-GPLink -Name $newGPOName -Target $ouDN -LinkEnabled Yes -Enforced No
Write-Host "✔ GPO linked to OU." -ForegroundColor Green

# ── Step 5: Verify ───────────────────────────────────────────────────────────
Get-GPInheritance -Target $ouDN | Select-Object -ExpandProperty GpoLinks
⚠️
If a parent GPO has Enforced set, your OU-level GPO will be overridden. To enforce your new GPO: Set-GPLink -Name $newGPOName -Target $ouDN -Enforced Yes
3
Disable the FIPS Flag in the OU GPO
Performed on the Domain Controller — sets FIPS = Disabled only for the workstation OU
🚨
Test immediately after this phase Before applying any cryptography policies (Phase 4), reboot the workstation, run gpupdate /force, and confirm the problem software works with FIPS disabled but no other changes applied yet. This establishes a clean baseline.
1
Open the GPO for editing
In GPMC, right-click FIPS-Exempt-Workstation-Policy → Edit…
2
Navigate to Security Options
Computer Configuration → Windows Settings → Security Settings → Local Policies → Security Options
3
Disable the FIPS policy
Scroll to "System cryptography: Use FIPS compliant algorithms for encryption, hashing, and signing". Double-click → check Define this policy setting → select Disabled → OK.
4
Add belt-and-suspenders registry preference
Navigate to: Computer Configuration → Preferences → Windows Settings → Registry Right-click → New → Registry Item:
  • Action: Update
  • Hive: HKEY_LOCAL_MACHINE
  • Key Path: SYSTEM\CurrentControlSet\Control\Lsa\FipsAlgorithmPolicy
  • Value name: Enabled  |  Type: REG_DWORD  |  Data: 0
5
Force policy refresh on the workstation
On the workstation, open an elevated command prompt and run: gpupdate /force then reboot. Verify FIPS is off: reg query HKLM\SYSTEM\CurrentControlSet\Control\Lsa\FipsAlgorithmPolicy /v Enabled — value should be 0x0.
PowerShell — Run on Domain Controller
$gpoName = "FIPS-Exempt-Workstation-Policy"
$gpo     = Get-GPO -Name $gpoName

# ── Method 1: GPO Registry Preference (Preferences node) ─────────────────────
Set-GPRegistryValue `
    -Guid      $gpo.Id `
    -Key       "HKLM\SYSTEM\CurrentControlSet\Control\Lsa\FipsAlgorithmPolicy" `
    -ValueName "Enabled" `
    -Type      DWord `
    -Value     0

# ── Method 2: Security Options INF (makes it appear in secpol.msc on workstation)
$domain     = "corp.example.com"
$sysvolPath = "\\$domain\SYSVOL\$domain\Policies\{$($gpo.Id)}\Machine\Microsoft\Windows NT\SecEdit"

if (-not (Test-Path $sysvolPath)) { New-Item $sysvolPath -ItemType Directory | Out-Null }

$inf = @"
[Unicode]
Unicode=yes
[Version]
signature="`$CHICAGO`$"
Revision=1
[System Access]
[Registry Values]
MACHINE\System\CurrentControlSet\Control\Lsa\FIPSAlgorithmPolicy\Enabled=4,0
"@
Set-Content -Path "$sysvolPath\GptTmpl.inf" -Value $inf -Encoding Unicode
Write-Host "✔ FIPS disabled in GPO." -ForegroundColor Green

# ── On the workstation: force GPO refresh ────────────────────────────────────
# Run this on the workstation after the above runs on the DC:
# gpupdate /force
# Restart-Computer -Force
✅
CHECKPOINT — Verify software works before Phase 4
Reboot the workstation, run gpupdate /force, and test the problem software thoroughly. It should now work with FIPS disabled and no other policy changes. Do not proceed to Phase 4 until this checkpoint passes. If the software still fails here, the issue is not solely the FIPS flag and further investigation is needed.
4
Incremental Cryptography Policy Rollout
Apply each group to the GPO on the DC — reboot workstation and test after every group
ℹ️
How this phase works Each group below is applied by running its PowerShell block on the Domain Controller to update the GPO. Then run gpupdate /force and reboot the workstation. Test the software. If it passes, move to the next group. If it breaks, use the rollback command shown in that group to remove just those settings.
Group A — AES & SHA384 (Additive Only)
Explicitly enables strong algorithms — adds nothing new to block
Very Low Risk
SettingValueWhy
AES 128/128 CipherEnabled = 1Guarantees AES is available after legacy ciphers are removed
AES 256/256 CipherEnabled = 1Strongest FIPS-approved symmetric cipher — must be explicit
SHA384 HashEnabled = 1Completes the SHA-2 family alongside SHA256/SHA512
PowerShell — Run on DC
$gpo  = Get-GPO -Name "FIPS-Exempt-Workstation-Policy"
$base = "HKLM\System\CurrentControlSet\Control\SecurityProviders\SCHANNEL"

foreach ($aes in @("AES 128/128", "AES 256/256")) {
    Set-GPRegistryValue -Guid $gpo.Id -Key "$base\Ciphers\$aes" -ValueName "Enabled" -Type DWord -Value 1
}
Set-GPRegistryValue -Guid $gpo.Id -Key "$base\Hashes\SHA384" -ValueName "Enabled" -Type DWord -Value 1
Write-Host "✔ Group A applied." -ForegroundColor Green
🔄 Rollback (if software breaks after this group) $gpo = Get-GPO -Name "FIPS-Exempt-Workstation-Policy"
foreach ($aes in @("AES 128/128","AES 256/256")) { Remove-GPRegistryValue -Guid $gpo.Id -Key "HKLM\...\Ciphers\$aes" -ValueName "Enabled" }
Remove-GPRegistryValue -Guid $gpo.Id -Key "HKLM\...\Hashes\SHA384" -ValueName "Enabled"
⟳ Reboot workstation & test after applying
Group B — DisabledByDefault Keys for Legacy Protocols
Belt-and-suspenders for protocols already set Enabled=0 in the baseline
Very Low Risk
Protocol / RoleValueWhy
SSL 2.0 Server & ClientDisabledByDefault = 1Some runtimes check only this key, not Enabled. Belt-and-suspenders alongside Enabled=0.
SSL 3.0 Server & ClientDisabledByDefault = 1
TLS 1.0 Server & ClientDisabledByDefault = 1
TLS 1.1 Server & ClientDisabledByDefault = 1
TLS 1.2 Server & ClientDisabledByDefault = 0
PowerShell — Run on DC
$gpo  = Get-GPO -Name "FIPS-Exempt-Workstation-Policy"
$base = "HKLM\System\CurrentControlSet\Control\SecurityProviders\SCHANNEL"

foreach ($p in @("SSL 2.0","SSL 3.0","TLS 1.0","TLS 1.1")) {
    foreach ($r in @("Server","Client")) {
        Set-GPRegistryValue -Guid $gpo.Id -Key "$base\Protocols\$p\$r" -ValueName "DisabledByDefault" -Type DWord -Value 1
    }
}
foreach ($r in @("Server","Client")) {
    Set-GPRegistryValue -Guid $gpo.Id -Key "$base\Protocols\TLS 1.2\$r" -ValueName "DisabledByDefault" -Type DWord -Value 0
}
Write-Host "✔ Group B applied." -ForegroundColor Green
🔄 Rollback foreach ($p in @("SSL 2.0","SSL 3.0","TLS 1.0","TLS 1.1","TLS 1.2")) { foreach ($r in @("Server","Client")) { Remove-GPRegistryValue -Guid $gpo.Id -Key "HKLM\...\Protocols\$p\$r" -ValueName "DisabledByDefault" } }
⟳ Reboot workstation & test after applying
Group C — Enable TLS 1.3
Adds the strongest available protocol; ignored on unsupported OS versions
Low Risk
SettingValueNotes
TLS 1.3 ServerEnabled = 1, DisabledByDefault = 0Requires Windows 10 1903+ or Server 2022. Silently ignored on older OS.
TLS 1.3 ClientEnabled = 1, DisabledByDefault = 0Test all outbound connections the application makes after applying.
PowerShell — Run on DC
$gpo  = Get-GPO -Name "FIPS-Exempt-Workstation-Policy"
$base = "HKLM\System\CurrentControlSet\Control\SecurityProviders\SCHANNEL"

foreach ($r in @("Server","Client")) {
    Set-GPRegistryValue -Guid $gpo.Id -Key "$base\Protocols\TLS 1.3\$r" -ValueName "Enabled"          -Type DWord -Value 1
    Set-GPRegistryValue -Guid $gpo.Id -Key "$base\Protocols\TLS 1.3\$r" -ValueName "DisabledByDefault" -Type DWord -Value 0
}
Write-Host "✔ Group C applied." -ForegroundColor Green
🔄 Rollback foreach ($r in @("Server","Client")) { Remove-GPRegistryValue -Guid $gpo.Id -Key "HKLM\...\Protocols\TLS 1.3\$r" -ValueName "Enabled"; Remove-GPRegistryValue ... -ValueName "DisabledByDefault" }
⟳ Reboot workstation & test after applying
Group D — Disable RC2 Cipher Family
Removes legacy RC2 variants not covered in the original script
Medium Risk
⚠️
RC2 is rare but may be used by custom or embedded components. Test any legacy integrations — COM objects, third-party DLLs, ODBC drivers — after this group.
SettingValue
RC2 128/128 CipherEnabled = 0
RC2 56/128 CipherEnabled = 0
RC2 40/128 CipherEnabled = 0
PowerShell — Run on DC
$gpo  = Get-GPO -Name "FIPS-Exempt-Workstation-Policy"
$base = "HKLM\System\CurrentControlSet\Control\SecurityProviders\SCHANNEL"

foreach ($rc2 in @("RC2 128/128", "RC2 56/128", "RC2 40/128")) {
    Set-GPRegistryValue -Guid $gpo.Id -Key "$base\Ciphers\$rc2" -ValueName "Enabled" -Type DWord -Value 0
}
Write-Host "✔ Group D applied." -ForegroundColor Green
🔄 Rollback foreach ($rc2 in @("RC2 128/128","RC2 56/128","RC2 40/128")) { Remove-GPRegistryValue -Guid $gpo.Id -Key "HKLM\...\Ciphers\$rc2" -ValueName "Enabled" }
⟳ Reboot workstation & test after applying
Group E — Minimum DH Key Size (2048-bit)
Prevents Logjam-style weak key negotiation
Medium Risk
⚠️
Any server or external service the workstation connects to that only supports 512 or 1024-bit DH will fail to negotiate TLS after this change. Carefully test all outbound connections — web services, APIs, database connections, directory services.
SettingValue
Diffie-Hellman ServerMinKeyBitLength2048
Diffie-Hellman ClientMinKeyBitLength2048
PowerShell — Run on DC
$gpo  = Get-GPO -Name "FIPS-Exempt-Workstation-Policy"
$kxKey = "HKLM\System\CurrentControlSet\Control\SecurityProviders\SCHANNEL\KeyExchangeAlgorithms\Diffie-Hellman"

Set-GPRegistryValue -Guid $gpo.Id -Key $kxKey -ValueName "ServerMinKeyBitLength" -Type DWord -Value 2048
Set-GPRegistryValue -Guid $gpo.Id -Key $kxKey -ValueName "ClientMinKeyBitLength" -Type DWord -Value 2048
Write-Host "✔ Group E applied." -ForegroundColor Green
🔄 Rollback Remove-GPRegistryValue -Guid $gpo.Id -Key $kxKey -ValueName "ServerMinKeyBitLength"
Remove-GPRegistryValue -Guid $gpo.Id -Key $kxKey -ValueName "ClientMinKeyBitLength"
⟳ Reboot workstation & test after applying
Group F — Cipher Suite Order Policy
Enforces strong AES-GCM cipher preference — highest compatibility risk
Higher Risk
🚨
Apply this group last and test most thoroughly. If the application or any of its dependencies hard-codes a cipher suite not in this list, TLS negotiation will fail entirely. Before applying, check if the application has its own TLS/cipher configuration file or registry settings.
#Cipher SuiteProtocol
1TLS_AES_256_GCM_SHA384TLS 1.3
2TLS_AES_128_GCM_SHA256TLS 1.3
3TLS_CHACHA20_POLY1305_SHA256TLS 1.3
4TLS_ECDHE_ECDSA_WITH_AES_256_GCM_SHA384TLS 1.2
5TLS_ECDHE_ECDSA_WITH_AES_128_GCM_SHA256TLS 1.2
6TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384TLS 1.2
7TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256TLS 1.2
8TLS_DHE_RSA_WITH_AES_256_GCM_SHA384TLS 1.2
9TLS_DHE_RSA_WITH_AES_128_GCM_SHA256TLS 1.2
PowerShell — Run on DC
$gpo     = Get-GPO -Name "FIPS-Exempt-Workstation-Policy"
$ciphers = "TLS_AES_256_GCM_SHA384,TLS_AES_128_GCM_SHA256,TLS_CHACHA20_POLY1305_SHA256," +
           "TLS_ECDHE_ECDSA_WITH_AES_256_GCM_SHA384,TLS_ECDHE_ECDSA_WITH_AES_128_GCM_SHA256," +
           "TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384,TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256," +
           "TLS_DHE_RSA_WITH_AES_256_GCM_SHA384,TLS_DHE_RSA_WITH_AES_128_GCM_SHA256"

Set-GPRegistryValue `
    -Guid      $gpo.Id `
    -Key       "HKLM\SOFTWARE\Policies\Microsoft\Cryptography\Configuration\SSL\00010002" `
    -ValueName "Functions" `
    -Type      String `
    -Value     $ciphers

Write-Host "✔ Group F applied." -ForegroundColor Green
🔄 Rollback Remove-GPRegistryValue -Guid $gpo.Id -Key "HKLM\SOFTWARE\Policies\Microsoft\Cryptography\Configuration\SSL\00010002" -ValueName "Functions"
⟳ Reboot workstation & test after applying
🏁
All groups applied — final verification
After all passing groups have been applied and the workstation is stable, run the following on the workstation to confirm the active configuration:

gpresult /h C:\GPOReport.html — generates a full GPO report.
Get-TlsCipherSuite | Select-Object Name | Format-Table — shows active cipher suites.
reg query "HKLM\SYSTEM\CurrentControlSet\Control\Lsa\FipsAlgorithmPolicy" — confirm FIPS = 0.

Document which groups were skipped (if any) as accepted risk exceptions and retain for your security audit records.
